# Backend E2E 测试名称参数透传修复设计

## 背景

DX CLI v0.1.119 在处理 `dx test e2e backend <path> -t "case name"` 时，能够从原始参数中识别测试名称，但会把 `-t` 直接追加到已配置命令的末尾。如果命令末尾包含 `# --workers...` 形式的 shell 注释，最终命令会类似：

```sh
vitest run '<path>' # --workers=1 -t 'case name'
```

shell 会忽略 `#` 后面的内容，导致 Vitest 收不到测试名称过滤参数并运行文件内全部用例。

## 目标

- 后端 E2E 的文件路径、`-t` 测试名称、`--` 后透传参数和默认 worker 参数都进入可执行命令，而不是落入注释。
- 带空格或 shell 特殊字符的测试名称继续使用现有转义规则完整透传。
- 保持 E2E 文件路径归一化、环境加载和默认 worker 值 8 不变。
- 保持 `requiresPath` 门禁：未指定文件或目录以及使用 `all` 时继续拒绝执行。

## 非目标

- 不重写通用 shell 命令解析器。
- 不修改目标工程的 `project.json` 或 `commands.json`。
- 不改变 unit test、E2E 环境选择或 worker 参数名称。
- 不调整现有命令配置格式。

## 根因

`handleTest` 通过字符串末尾拼接构造 E2E 命令：

1. 从 `fileCommand` 或 Nx target 的 `options.command` 得到基础命令。
2. 追加测试路径。
3. 追加 `-t` 和测试名称。
4. 追加 `--` 后的透传参数。
5. 追加默认 `--workers=8`。

该流程没有区分命令的可执行部分和行尾 shell 注释。同时，worker 检测会扫描整条字符串，使注释中的 `--workers` 被误判为有效参数。

## 方案比较

### 方案 A：测试命令使用注释感知的参数追加（采用）

增加一个小型辅助函数，将命令分成“可执行部分”和“行尾注释”，把新增参数插入两者之间。注释边界只识别引号外、位于行首或空白后的 `#`，避免把引号内或普通参数中的 `#` 当作注释。

优点：统一解决路径、测试名称、透传参数和 worker 的同类问题；保持注释与输出可读性；改动集中在 test 命令拼接。

风险：需要维护一个有限状态扫描，但范围仅覆盖识别 shell 注释边界，不承担完整 shell 解析。

### 方案 B：只匹配 `# --workers` 并移动 `-t`

优点：代码改动最少。

缺点：只解决当前字符串形式；路径、透传参数和其他行尾注释仍可能被截断；正则容易误判引号中的 `#`。

### 方案 C：删除目标工程命令中的注释

优点：无需修改 DX。

缺点：只是绕过问题，其他目标工程仍会复现，无法形成 CLI 回归保障。

## 设计

在 `lib/cli/commands/core.js` 内增加两个局部能力：

1. 定位命令中第一个有效的 shell 注释边界。
   - 扫描单引号、双引号和反斜杠转义状态。
   - 仅当 `#` 不在引号内，且位于字符串开头或前一个字符为空白时，视为注释开始。
2. 将一个或多个已完成 shell 转义的参数追加到可执行部分末尾，并把原注释放回末尾。

E2E 命令按以下顺序构造：

```text
基础命令 -> 测试路径 -> -t 测试名称 -> 透传参数 -> 默认 worker -> 行尾注释
```

对于包含 `{TEST_PATH}` 的 `fileCommand`，占位符替换逻辑保持不变；后续参数使用注释感知追加。对于解析出的 Nx direct target，测试路径也使用同一追加逻辑，避免 direct command 自带注释时路径被截断。

worker 检测仅检查命令的可执行部分。注释中的 `--workers` 不再算作显式配置；没有有效 worker 参数时仍追加现有默认值 `--workers=8`。已存在的有效 `--workers` 和用户通过 `--` 传入的 worker 参数继续优先，不重复追加默认值。

## 错误处理与兼容性

- 不新增用户可见错误类型。
- 没有注释的命令输出保持原有参数顺序和语义。
- 无法形成有效命令时沿用现有配置校验和执行错误处理。
- `validateTestPositionals` 和 `requiresPath` 逻辑不修改。
- 环境标志、dotenv 层加载、cwd 和路径归一化逻辑不修改。

## 测试设计

在 `test/test-command-e2e-targets.test.js` 增加 CLI 级回归测试，构造一个带行尾 `# --workers=1` 注释的 Nx direct target 命令，并通过真实 shell 执行验证：

- 带空格的 `-t "case name"` 被底层进程完整接收。
- 归一化后的测试路径被底层进程接收。
- 默认 `--workers=8` 被底层进程接收。
- 日志中的完整命令显示动态参数位于注释之前。

继续运行 `test/test-command-e2e-guard.test.js`，确认以下门禁未回归：

- `dx test e2e backend` 被拒绝。
- `dx test e2e backend all` 被拒绝。
- `dx test e2e all` 被拒绝。

最后运行完整 `pnpm test`，确认其他 CLI 行为不受影响。

## 验收映射

- 单用例执行：真实 shell 回归测试验证底层 runner 收到完整 `-t` 值。
- 命令未被截断：断言命令顺序和底层进程实际参数。
- 带空格名称：固定使用 `case name` 覆盖。
- 文件/目录门禁：保留并执行现有 guard 测试。
